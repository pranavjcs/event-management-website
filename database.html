Database Design (3NF)

The database is normalized to 3NF to remove redundancy and ensure referential integrity.

Tables and Attributes

  • STUDENT (student_id PK, full_name, email UNIQUE, department, password_hash, created_at)
  • ADMIN (admin_id PK, full_name, email UNIQUE, password_hash, created_at)
  • EVENT (event_id PK, admin_id FK, title, category, event_date, event_time, venue, seats, description)
  • REGISTRATION (registration_id PK, event_id FK, student_id FK, registered_at)

SQL CREATE TABLE

CREATE TABLE STUDENT (
  student_id VARCHAR(20) PRIMARY KEY,
  full_name VARCHAR(100) NOT NULL,
  email VARCHAR(120) NOT NULL UNIQUE,
  department VARCHAR(80) NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE ADMIN (
  admin_id VARCHAR(20) PRIMARY KEY,
  full_name VARCHAR(100) NOT NULL,
  email VARCHAR(120) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE EVENT (
  event_id VARCHAR(20) PRIMARY KEY,
  admin_id VARCHAR(20) NOT NULL,
  title VARCHAR(150) NOT NULL,
  category VARCHAR(60) NOT NULL,
  event_date DATE NOT NULL,
  event_time VARCHAR(20) NOT NULL,
  venue VARCHAR(100) NOT NULL,
  seats INT NOT NULL,
  description TEXT NOT NULL,
  CONSTRAINT fk_event_admin FOREIGN KEY (admin_id)
    REFERENCES ADMIN(admin_id)
);

CREATE TABLE REGISTRATION (
  registration_id VARCHAR(20) PRIMARY KEY,
  event_id VARCHAR(20) NOT NULL,
  student_id VARCHAR(20) NOT NULL,
  registered_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_reg_event FOREIGN KEY (event_id)
    REFERENCES EVENT(event_id),
  CONSTRAINT fk_reg_student FOREIGN KEY (student_id)
    REFERENCES STUDENT(student_id)
);

ER Diagram (Text Representation)

ADMIN (1) --- creates --- (N) EVENT
STUDENT (1) --- registers --- (N) REGISTRATION --- belongs to --- (1) EVENT 

ADMIN ||--o{ EVENT : creates
STUDENT ||--o{ REGISTRATION : registers
EVENT ||--o{ REGISTRATION : has

One admin creates many events. One student can register for many events, and each event can have many registrations. REGISTRATION is the associative entity resolving the many-to-many relationship.
